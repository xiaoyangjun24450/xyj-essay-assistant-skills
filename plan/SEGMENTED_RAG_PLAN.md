# 分段检索式改写流程说明
本文档说明当前分段检索式改写流程的实现思路，重点描述每个阶段的输入、输出，以及 LLM 在流程中的职责。

## 一句话概括
整体流程采用“模板预处理 -> 模板语义分段 -> 分段校验 -> 生成 chunk -> 素材预处理与分段 -> 按 chunk 检索素材 -> 逐块改写 -> 文档还原”的方式完成论文改写。

## 核心原则
- 全流程围绕段落级唯一标识展开，确保分段、改写、校验、还原可以对齐。
- 先理解模板结构，再切成可控 chunk，再为每个 chunk 检索最相关素材，最后逐块改写。
- 语义边界优先于字数边界，字数约束只在必要时作为二次拆分条件。
- 模板和素材分别建模，只在 chunk 级别发生匹配，避免上下文浪费。
- 每个阶段都要输出可检查的中间结果，便于回溯和修复。

## 总体流程
```text
模板预处理
  -> 模板语义分段
  -> 分段结果校验/修复
  -> 基于分段生成 chunk
  -> 素材预处理与语义分段（可选）
  -> 按 chunk 检索相关素材（可选）
  -> 逐块改写与结果修复
  -> 文档还原
```

## 1. 模板预处理
目标是把 docx 模板转换成后续流程可处理的“带段落锚点的全文文本”。

输入：
- 模板文档。

处理方式：
- 提取正文段落。
- 为每个段落建立稳定的段落 ID。
- 将原始格式信息保留下来，供最终还原使用。
- 将正文整理成“每行一个段落”的统一文本表示。

输出：
- 带段落 ID 的全文文本。
- 与段落对应的格式映射信息。
- 可用于最终还原的原始文档结构副本。

LLM 参与：
- 不参与。

## 2. 模板语义分段
目标是识别模板内部的自然结构单元，例如背景、方法、实验、结论等语义段。

输入：
- 带段落 ID 的模板全文文本。
- 当前窗口中的段落内容。
- 必要的前后文提示。
- 上一个窗口最后一个分段的信息。

处理方式：
- 全文不会一次性交给 LLM，而是按窗口分批处理。
- 每个窗口都要求 LLM 在不打乱段落顺序的前提下完成语义分段。
- 如果某个分段跨窗口延续，下一窗口需要先承接上一窗口的尾部分段。

发给 LLM 的内容：
- 当前窗口内的 `[段落ID] + 段落文本`。
- 当前窗口在全文中的位置。
- 上一窗口最后一个分段的概要与段落范围。
- 窗口外前后文提示。
- 分段规则与输出格式要求。

LLM 输出：
- 一个 segment 列表。
- 每个 segment 至少包含：
  - `segment_id`
  - `summary`
  - `purpose`
  - `para_ids`

输出到下一阶段：
- 模板语义分段结果。

## 3. 分段结果校验与修复
目标是确保模板分段结果足够可靠，能够安全进入 chunk 生成阶段。

输入：
- 模板全文段落列表。
- 上一阶段得到的分段结果。

处理方式：
- 检查段落是否遗漏、重复、乱序。
- 检查分段集合是否完整覆盖全文。
- 能用规则修正的问题优先本地修正。
- 无法直接判断归属的缺失段落，再交给 LLM 做插回决策。

发给 LLM 的内容：
- 当前已有的分段结果。
- 待修复段落本身。
- 待修复段落的前后邻居。
- 候选可插入的 segment 信息。
- 可执行动作约束，例如“并入已有 segment”或“新建 segment”。

LLM 输出：
- 针对每个待修复段落的决策列表。
- 每条决策通常包括：
  - 该段落应该并入哪个已有 segment
  - 或者是否应新建一个 segment
  - 新建时对应的摘要与用途标签

输出到下一阶段：
- 校验并修复后的模板分段结果。

## 4. 基于分段生成 chunk
目标是把模板分段结果转换成适合改写的最小工作单元。

输入：
- 模板全文段落列表。
- 已校验的模板分段结果。

处理方式：
- 优先按 segment 生成 chunk。
- 如果单个 segment 过长，再在该 segment 内继续拆分。
- chunk 拆分不打断段落顺序，也不脱离所属 segment 的语义上下文。

输出：
- 一组 chunk 文本。
- 每个 chunk 的元数据，包括：
  - 所属 segment
  - 段落范围
  - 是否为拆分片段
  - 在所属 segment 中的相对位置

LLM 参与：
- 不参与。

## 5. 素材预处理与语义分段
这是可选阶段，仅在用户提供素材时启用。目标是把用户素材也转换成可检索的语义单元。

输入：
- 用户提供的 docx、txt、md 等素材。

处理方式：
- 先把每份素材整理成统一的段落化文本。
- 为素材段落建立稳定编号。
- 对每份素材分别进行语义分段。
- 最后将多份素材合并成统一素材池。

发给 LLM 的内容：
- 当前素材窗口中的 `[素材段落编号] + 段落文本`。
- 窗口位置与前后文提示。
- 上一个窗口最后一个素材分段的信息。
- 素材分段规则与输出格式要求。

LLM 输出：
- 素材 segment 列表。
- 每个素材 segment 至少包含：
  - `segment_id`
  - `summary`
  - `purpose`
  - `para_ids`

输出到下一阶段：
- 编号后的素材全文。
- 素材语义分段结果。

## 6. 按 chunk 检索相关素材
这是可选阶段，仅在存在素材池时启用。目标是为每个模板 chunk 找到最相关的少量素材，并生成该 chunk 的改写大纲。

输入：
- 当前模板 chunk。
- 当前 chunk 的元数据。
- 用户改写需求。
- 素材分段结果。
- 编号后的素材全文。

处理方式：
- 不把全部素材全文直接交给 LLM，而是先以素材 segment 级信息做筛选。
- LLM 根据 chunk 的主题、用途、所在 section 位置，选择最相关的素材段落。
- 同时产出该 chunk 的局部写作大纲。

发给 LLM 的内容：
- 当前 chunk 原文。
- 当前 chunk 的语义信息和位置属性。
- 用户改写任务。
- 全部素材 segment 的摘要、用途、段落范围及简要预览。
- 检索数量限制与输出格式要求。

LLM 输出：
- 当前 chunk 对应的素材段落编号列表。
- 当前 chunk 的改写大纲。
- 选择这些素材的理由。

输出到下一阶段：
- chunk 级素材检索结果。
- chunk 级改写大纲。

## 7. 逐块改写与结果修复
目标是在严格保留结构约束的前提下完成逐块改写。

输入：
- 当前 chunk 原文。
- 用户改写需求。
- 当前 chunk 的语义信息。
- 检索到的相关素材。
- 当前 chunk 的改写大纲。

处理方式：
- 每个 chunk 独立改写。
- 改写后先做结构校验，不满足约束时进入修复流程。
- 只有通过校验的结果才会进入最终还原。

发给 LLM 的内容：
- 改写需求。
- 当前 chunk 原文。
- 当前 chunk 的语义信息。
- 相关素材原文。
- 当前 chunk 的写作大纲。
- 对结构、段落锚点、格式标记、公式合法性的约束要求。

LLM 输出：
- 改写后的 chunk 文本。
- 文本仍需保留原有段落锚点和整体结构。

修复阶段发给 LLM 的内容：
- 校验失败的错误信息。
- 原始 chunk 内容。
- 待修复的改写结果。
- 修复约束。

修复阶段 LLM 输出：
- 修复后的完整 chunk 文本。

输出到下一阶段：
- 全部通过校验的改写 chunk。

## 8. 文档还原
目标是将改写后的 chunk 重新映射回原始文档结构，生成最终 docx。

输入：
- 改写后的 chunk 集合。
- 模板预处理阶段保留的格式映射信息。
- 原始文档结构副本。

处理方式：
- 按段落 ID 把新内容写回原始文档对应段落。
- 将中间表示的格式标记还原成文档样式。
- 将中间表示的公式还原成文档可用形式。
- 尽量保持非正文结构、版式和可保留元素不变。

输出：
- 最终 docx 文档。

LLM 参与：
- 不参与。

## 这套方案解决的问题
- 解决纯字数切块带来的语义割裂问题。
- 解决素材一次性全量注入导致的上下文浪费问题。
- 解决改写任务缺乏局部目标、导致段间风格漂移的问题。
- 解决改写结果难以回溯、难以校验、难以修复的问题。

## 适用边界
- 适合有明确模板结构、需要保持格式稳定的论文或报告类文档。
- 适合用户能够提供补充素材，希望提升改写针对性的场景。
- 不适合强依赖表格、复杂图形重写或大规模版式重排的场景。
